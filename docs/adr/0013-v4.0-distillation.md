# ADR-0013: v4.0 distillation — prune broken, consolidate overlaps

Date: 2026-05-04
Status: accepted

## Context

By v3.8 the plugin had grown organically to 36 skills + 9 agents + 8 rules + 7 hooks + 15 validators + 3 KB agents. A diagnostic on 2026-05-04 (see `docs/sessions/2026-05-04-v4.0-distillation.md` and the corresponding plan) identified four classes of debt:

1. **Broken in practice**: graphify's three blocking Windows issues ([safishamsi/graphify#378](https://github.com/safishamsi/graphify/issues/378), [#244](https://github.com/safishamsi/graphify/issues/244), [#501](https://github.com/safishamsi/graphify/issues/501)) made the "dual-engine" narrative fiction on the operator's primary machine. The bootstrap script always marked graphify BROKEN; codebase-memory-mcp (CBM) carried 100% of the load.

2. **Documentation lies**: PRD's `Last reviewed` was three minor versions stale; SPEC claimed "9 cases" of validators while the suite shipped 15; the subagent destinations table in DELEGATION-RULE was missing the three KB agents.

3. **Dead weight**: the `simplify-ignore` hook was authored but never registered in `hooks/hooks.json` and never invoked in production. The `notion-kb-workflow` skill was deprecated in ADR-0012 but its directory was never removed. The `/code-simplify` command's loose entry point produced bad results because it lacked scope and readiness checks (the skill itself was sound).

4. **Misleading enforcement language**: 6 of 8 rules titled their enforcement section "Inviolable rules" while having no runtime hook backing them — only the two authoring gates (`pre-write-skill-gate.sh`, `pre-write-agent-gate.sh`) enforced their respective rules at the tool-call layer. Consumers had no way to tell which rules were gated and which were context-only.

5. **Hidden coupling**: `setup-rules.sh --all` chained automatically into `setup-code-graph.sh`, so a project that only wanted rule symlinks received the entire engine install as a side effect.

The plugin is maintained by one developer for a small number of projects. Each accreted feature was justified at the time it shipped, but the aggregate complexity cost had begun to outweigh the benefit. Re-shipping with broken engines, lying docs, and unused hooks erodes the contract the plugin makes with consumers.

## Decision

Ship v4.0 as a **prune + consolidate** pass — surgically remove what doesn't work, fix what lies, and clarify what was ambiguous, without redesigning the architecture. The slices:

1. **Deprecate graphify; single-engine code-graph (codebase-memory-mcp)**.
   - Remove `install_graphify()` from `setup-code-graph.sh` and its associated pin/status/version variables.
   - Simplify `check-code-graph-engines.sh` to read only the CBM key.
   - Rewrite `skills/code-graph/SKILL.md`, the integrations rule, and the slash command to single-engine.
   - Update `agents/code-reviewer.md` and `agents/security-auditor.md` Step 0.5 to reference CBM only.
   - Update validator 07 to drop graphify-positive assertions; keep the `graphify claude install` prohibition checks (the upstream command still exists as a footgun).
   - Update validator 08 to drop the `GRAPHIFY_PIN` assertion.
   - Update e2e scenario 01 to use `--skip-cbm` (the only remaining engine flag).
   - Preserve a `graphify` key in the state file marked `DEPRECATED` for backward compat with cached older versions of `check-code-graph-engines.sh`.
   - Bump plugin version to `4.0.0`.

2. **Fix `/code-simplify`; delete `simplify-ignore` hook**.
   - Add explicit scope (Step 0), test-readiness verification (Step 0.5), and stronger convention guidance (Step 1) to the slash command.
   - Delete `hooks/simplify-ignore.sh`, `hooks/simplify-ignore-test.sh`, `hooks/SIMPLIFY-IGNORE.md` — never registered, never used, Windows-incompatible.

3. **Fix stale docs**.
   - PRD `Last reviewed` → `2026-05-04 (v4.0)`; add v3.8 and v4.0 roadmap entries.
   - SPEC `Last reviewed` → `2026-05-04 (v4.0)`; correct "9 cases" → "15 cases (01–15)" with the full list; rewrite Layer 8 for single-engine CBM.
   - DELEGATION-RULE: add `kb-pipeline`, `kb-curator`, `kb-backfiller` to the subagent destinations table.

4. **Remove `notion-kb-workflow/`**.
   - Delete the directory entirely; replace the multi-line CLAUDE.md section with a one-line note pointing at `kb-pipeline`.

5. **Decouple `setup-rules.sh` from code-graph**.
   - Remove the auto-chain to `setup-code-graph.sh`. Add an info message at the end pointing at the separate command.

6. **Add `enforcement` metadata to rules**.
   - Frontmatter field `enforcement: hook` for the two authoring gates (with `hook:` pointing at the script). `enforcement: context-only` for the other six rules. Document the field in `rules/_meta/rule-template.md` and `rules/README.md`.

7. **Fix `setup-rules.sh` Windows compatibility**.
   - Replace the bare `ln -s` failure path with a fallback: on Windows where Developer Mode is off, fall back to `cp` with a warning that copies do not auto-update on plugin upgrades.

The slices are independent enough to be reviewed and reverted individually. They land as one PR per slice on the `chore/v4.0-distillation` branch.

## Decisions explicitly NOT in scope

- `research-first-dev` and `source-driven-development` stay separate — analysis showed orthogonal triggers (per-import vs per-pattern).
- The KB pipeline (`kb-pipeline` + `kb-curator` + `kb-backfiller`) stays as-is — the architecture is sound; it needs vault content over time, not redesign.
- Authoring gates (`batuta-skill-authoring`, `batuta-agent-authoring`, `batuta-rule-authoring`) stay — they prevent the exact hardcoding/over-fitting problems that motivated this distillation.
- `intent-capture` stays — same reason; vague directives are the failure mode the plugin must guard against.

## Consequences

### What we gain

- **Honesty**: the docs match reality. Validator counts, version numbers, subagent tables, and engine narratives are all correct.
- **Smaller attack surface**: ~700 fewer lines of unused hook code (`simplify-ignore`), one fewer external dependency (graphify on PyPI), one fewer state-file key to maintain.
- **Clarity for consumers**: the new `enforcement:` frontmatter lets a project that imports a rule know whether it gets a runtime guarantee or context-only guidance.
- **Smoother Windows experience**: with graphify gone, the cross-platform story is uniform; the `setup-rules.sh` copy fallback eliminates the Developer Mode hard-fail.
- **Faster `/code-simplify`**: explicit scope and readiness checks stop the command from operating blind.

### What we give up

- **Multimodal graph queries** (PDFs, images, docs as first-class graph nodes). graphify supported these; CBM does not. No Batuta project has used them in production. If they become necessary later, graphify can be re-introduced when its Windows issues are resolved upstream — the contract is not closed against future re-introduction, just simplified for now.
- **The `--mcp` slash flag** for code-graph (was graphify-only; CBM is natively MCP).

### Backward compatibility

- The state file at `~/.claude/code-graph-engines.json` keeps its `graphify` key (now always `"status": "DEPRECATED"`). Cached older versions of `check-code-graph-engines.sh` continue to parse the file without error.
- ADR-0007 (the original dual-engine ADR) is preserved as a historical record; it is not edited. Reading the two ADRs side by side gives the full evolution.
