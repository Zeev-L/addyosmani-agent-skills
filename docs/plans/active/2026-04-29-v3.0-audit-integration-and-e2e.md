### Slice v3.0b — audit-chain × code-graph integration

Today: `code-reviewer` and `security-auditor` Step 0 check `git diff` and either return NOT APPLICABLE (clean tree) or proceed to a 5-axis / scope review that reads files individually.

This slice adds **Step 0.5** between the not-applicable check and the framework review: if the code-graph is available (`~/.claude/code-graph-engines.json` reports at least one engine OK), consult it to enumerate **all files transitively related to the diff** (call graph neighbors + symbol references). The reviewer reads those too, not just the diff.

Concrete:

- [agents/code-reviewer.md](agents/code-reviewer.md): add **Step 0.5 — Blast-radius enumeration via code-graph**. Pseudocode:
  ```
  if code-graph available (check-code-graph-engines.sh exit 0):
    for each modified file in diff:
      query code-graph for: callers, callees, symbol-references
    blast_radius = union of all returned files
    Read every file in blast_radius (not just the diff hunks)
  else:
    log "code-graph unavailable; falling back to diff-only review"
    proceed with diff-only review
  ```
  Step 0.5 is **non-blocking**: it never returns BLOCKED, just enriches the context the auditor reads. Existing 5-axis framework runs unchanged.
- [agents/security-auditor.md](agents/security-auditor.md): same pattern, but the blast-radius query focuses on auth/data-flow paths (e.g. trace_call_path from any modified function back to network/DB/disk boundaries).
- [agents/test-engineer.md](agents/test-engineer.md): **NOT modified** — its leverage is on tests, not blast radius. Keeping its scope tight is intentional.
- [docs/adr/0008-audit-chain-code-graph-integration.md](docs/adr/0008-audit-chain-code-graph-integration.md), new: documents the rationale, why graceful-degrade on missing engine, why `test-engineer` is excluded.
- [tests/v2.5-validators/09-audit-chain-graph-integration.sh](tests/v2.5-validators/09-audit-chain-graph-integration.sh), new: validates that `code-reviewer.md` and `security-auditor.md` declare Step 0.5 with the graceful-degrade clause, and that `test-engineer.md` does NOT (scope guard).

### Slice v3.0c — E2E test harness

Up to v2.9 the plugin's tests are static contract validators (grep against agent prompts). This slice adds end-to-end behavior tests that drive an actual `claude` CLI session against a sandbox repo and verify outcomes.

- [tests/e2e/](tests/e2e/), new directory:
  - `run.sh` — orchestrator. Spins up a temp sandbox, runs each scenario, reports.
  - `scenarios/01-fresh-project-init.sh` — `mkdir sandbox && cd sandbox && claude --print --model sonnet "<prompt>"` where prompt simulates "I just started a Python project". Verifies the resulting tree contains `CLAUDE.md`, `docs/PRD.md`, `docs/SPEC.md`, `docs/plans/active/`, `docs/sessions/`, `.gitignore`. Asserts the auto-trigger of `batuta-project-hygiene mode=project-init` happened.
  - `scenarios/02-research-first-citation.sh` — sandbox already has a `pyproject.toml` with `fastapi`. Prompt: "add a hello-world FastAPI endpoint to src/api.py". Verify the resulting file contains a `# Source: ` citation comment for the FastAPI import.
  - `scenarios/03-code-graph-trigger.sh` — sandbox with `~5k LOC of Python. Prompt: "explain the architecture". Verify the response references `[via graphify]` or `[via codebase-memory-mcp]` (i.e., the skill auto-triggered).
  - `scenarios/04-audit-chain-clean-tree.sh` — clean working tree. Prompt asks for review. Verify the `code-reviewer` agent returns the literal `AUDIT RESULT: NOT APPLICABLE` string.
  - `README.md` — describes how to run the harness, the model used (`claude --model sonnet`), prerequisites (`claude` CLI installed and authenticated), and how to interpret the output.

Each scenario is a bash script returning 0 on pass / non-zero on fail. The `run.sh` aggregates results.

Acceptance: `bash tests/e2e/run.sh` returns 0 (all pass) on this machine. The harness does NOT block the static-validator suite — it's a separate, slower suite that the operator runs manually before tagging a release.
