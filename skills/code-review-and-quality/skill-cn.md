# 代码审查和质量

## 概述

具有质量门的多维代码审查。每个更改在合并之前都经过审查——没有例外。审查涵盖五个轴：正确性、可读性、架构、安全性和性能。

**批准标准：** 当更改肯定改善整体代码健康时批准更改，即使它不完美。完美的代码不存在——目标是持续改进。不要因为更改不完全是你编写它的方式而阻止更改。如果它改善了代码库并遵循项目的约定，就批准它。

## 使用场景

- 在合并任何 PR 或更改之前
- 完成功能实现后
- 当另一个代理或模型生成了你需要评估的代码时
- 重构现有代码时
- 任何错误修复后（审查修复和回归测试）

## 五轴审查

每次审查都从这些维度评估代码：

### 1. 正确性

代码是否做了它声称要做的事情？

- 是否符合规范或任务要求？
- 是否处理了边缘情况（null、空、边界值）？
- 是否处理了错误路径（不仅仅是快乐路径）？
- 是否通过了所有测试？测试是否真正测试了正确的事情？
- 是否有差一错误、竞争条件或状态不一致？

### 2. 可读性和简洁性

另一个工程师（或代理）是否可以在没有作者解释的情况下理解此代码？

- 名称是否具有描述性并符合项目约定？（没有不带上下文的 `temp`、`data`、`result`）
- 控制流是否简单（避免嵌套三元、深度回调）？
- 代码是否逻辑组织（相关代码分组、清晰的模块边界）？
- 是否有任何应该简化的"聪明"技巧？
- **这可以用更少的行完成吗？**（1000 行而 100 行就足够是失败）
- **抽象是否值得其复杂性？**（在第三个用例之前不要泛化）
- 注释是否有助于澄清非显而易见的意图？（但不要注释明显的代码。）
- 是否有死代码工件：无操作变量（`_unused`）、向后兼容填充或 `// removed` 注释？

### 3. 架构

更改是否适合系统的设计？

- 是否遵循现有模式或引入新模式？如果是新的，是否合理？
- 是否保持干净的模块边界？
- 是否有应该共享的代码重复？
- 依赖关系是否流向正确的方向（没有循环依赖）？
- 抽象级别是否适当（不是过度工程，不是太耦合）？

### 4. 安全性

有关详细的安全指导，请参见 `security-and-hardening`。更改是否引入了漏洞？

- 用户输入是否经过验证和净化？
- 秘密是否远离代码、日志和版本控制？
- 是否检查了需要的身份验证/授权？
- SQL 查询是否参数化（无字符串连接）？
- 输出是否编码以防止 XSS？
- 依赖项是否来自可信来源且没有已知漏洞？
- 来自外部源（API、日志、用户内容、配置文件）的数据是否被视为不可信？
- 外部数据流是否在使用于逻辑或渲染之前在系统边界进行验证？

### 5. 性能

有关详细的性能分析和优化，请参见 `performance-optimization`。更改是否引入了性能问题？

- 有任何 N+1 查询模式吗？
- 有任何无界循环或不受约束的数据获取吗？
- 有任何应该是异步的同步操作吗？
- UI 组件中有任何不必要的重新渲染吗？
- 列表端点上缺少分页吗？
- 热路径中创建的任何大对象？

## 更改大小

小且集中的更改更容易审查、更快合并且更安全地部署。目标这些大小：

```
~100 行更改   → 好。一次坐下来可审查。
~300 行更改   → 如果它是单个逻辑更改，可以接受。
~1000 行更改  → 太大。拆分它。
```

**什么算作"一个更改"：** 一个独立的修改，解决一件事，包括相关测试，并在提交后保持系统功能。一个功能的一部分——不是整个功能。

**当更改太大时的拆分策略：**

| 策略 | 如何 | 何时 |
|----------|-----|------|
| **堆叠** | 提交一个小更改，基于它开始下一个 | 顺序依赖 |
| **按文件组** | 对需要不同审查者的组进行单独更改 | 横切关注点 |
| **水平** | 首先创建共享代码/存根，然后创建消费者 | 分层架构 |
| **垂直** | 拆分为功能的较小全栈切片 | 功能工作 |

**当大更改可接受时：** 完整的文件删除和自动化重构，审查者只需要验证意图，而不是每一行。

**将重构与功能工作分开。** 重构现有代码和添加新行为的更改是两个更改——分别提交它们。小型清理（变量重命名）可以根据审查者的判断包括在内。

## 更改描述

每个更改都需要一个在版本控制历史中独立存在的描述。

**第一行：** 简短、祈使句、独立。 "删除 FizzBuzz RPC" 而不是 "正在删除 FizzBuzz RPC。" 必须足够 informative，以便搜索历史的人可以在不阅读差异的情况下理解更改。

**正文：** 正在更改的内容和原因。包括代码中本身不可见的上下文、决策和推理。在相关时链接到错误编号、基准测试结果或设计文档。当存在时承认方法缺点。

**反模式：** "修复错误，" "修复构建，" "添加补丁，" "从 A 移动代码到 B，" "阶段 1，" "添加便利功能。"

## 审查流程

### 步骤 1：理解上下文

在查看代码之前，理解意图：

```
- 这个更改试图完成什么？
- 它实现了什么规范或任务？
- 预期的行为更改是什么？
```

### 步骤 2：首先审查测试

测试揭示意图和覆盖范围：

```
- 是否为更改存在测试？
- 它们是否测试行为（不是实现细节）？
- 是否覆盖了边缘情况？
- 测试是否有描述性名称？
- 如果代码更改，测试会捕获回归吗？
```

### 步骤 3：审查实现

牢记五个轴遍历代码：

```
对于更改的每个文件：
1. 正确性：此代码是否做了测试说它应该做的事情？
2. 可读性：我可以在没有帮助的情况下理解这个吗？
3. 架构：这适合系统吗？
4. 安全性：有任何漏洞吗？
5. 性能：有任何瓶颈吗？
```

### 步骤 4：分类发现

用其严重性标记每个注释，以便作者知道什么是必需的 vs 可选的：

| 前缀 | 含义 | 作者操作 |
|--------|---------|---------------|
| *（无前缀）* | 必需更改 | 合并前必须解决 |
| **关键：** | 阻止合并 | 安全漏洞、数据丢失、破坏的功能 |
| **细节：** | 轻微，可选 | 作者可以忽略——格式、样式偏好 |
| **可选：** / **考虑：** | 建议 | 值得考虑但不是必需 |
| **仅供参考** | 仅信息 | 不需要操作——供将来参考的上下文 |

这防止作者将所有反馈视为强制性并浪费时间在可选建议上。

### 步骤 5：验证验证

检查作者的验证故事：

```
- 运行了什么测试？
- 构建通过了吗？
- 是否手动测试了更改？
- UI 更改是否有截图？
- 是否有前后比较？
```

## 多模型审查模式

使用不同的模型进行不同的审查视角：

```
模型 A 编写代码
    │
    ▼
模型 B 审查正确性和架构
    │
    ▼
模型 A 处理反馈
    │
    ▼
人类做出最终决定
```

这捕获单个模型可能遗漏的问题——不同的模型有不同的盲点。

**审查代理的示例提示：**
```
审查此代码更改的正确性、安全性和对我们项目约定的遵守情况。
规范说 [X]。更改应该 [Y]。
将任何问题标记为关键、重要或建议。
```

## 死代码卫生

任何重构或实现更改后，检查孤立代码：

1. 识别现在无法访问或未使用的代码
2. 明确列出它
3. **在删除之前询问：** "我应该删除这些现在未使用的元素：[列表]？"

不要留下死代码——它会混淆未来的读者和代理。但不要默默地删除你不确定的事情。如有疑问，询问。

```
死代码识别：
- formatLegacyDate() 在 src/utils/date.ts 中——被 formatDate() 替换
- OldTaskCard 组件在 src/components/ 中——被 TaskCard 替换
- LEGACY_API_URL 常量在 src/config.ts 中——没有剩余引用
→ 可以安全删除这些吗？
```

## 审查速度

缓慢的审查会阻塞整个团队。审查上下文切换的成本小于强加给他人的等待成本。

- **在一个工作日内回应** —— 这是最大值，不是目标
- **理想节奏：** 在审查请求到达后不久回应，除非深度专注于编码。典型的更改应该在同一天内完成多个审查轮次
- **优先考虑快速个人回应** 而不是快速最终批准。即使需要多轮，快速反馈也会减少挫折感
- **大更改：** 要求作者拆分它们，而不是审查一个巨大的变更集

## 处理分歧

解决审查争议时，应用此层次结构：

1. **技术事实和数据** 覆盖意见和偏好
2. **样式指南** 是样式问题的绝对权威
3. **软件设计** 必须根据工程原则评估，而不是个人偏好
4. **代码库一致性** 如果不会降低整体健康度，则可以接受

**不要接受"我稍后清理它。"** 经验表明推迟的清理很少发生。要求在提交之前进行清理，除非是真正的紧急情况。如果周围环境不能在此更改中解决，则要求提交带有自我分配的 bug。

## 审查中的诚实

审查代码时——无论是由你、另一个代理还是人类编写的：

- **不要橡皮图章。** 没有审查证据的"LGTM"对任何人都没有帮助。
- **不要淡化真正的问题。** "这可能是一个小问题" 当它是会打击生产的错误时是不诚实的。
- **尽可能量化问题。** "此 N+1 查询将为列表中的每项添加约 50 毫秒" 比 "这可能很慢" 更好。
- **推回有明显问题的方法。** 奉承是审查中的失败模式。如果实现有问题，直接说出来并提议替代方案。
- **优雅地接受覆盖。** 如果作者有完整的上下文并不同意，遵循他们的判断。评论代码，而不是人——将个人批评重新构建为专注于代码本身。

## 依赖纪律

代码审查的一部分是依赖项审查：

**在添加任何依赖项之前：**

1. 现有堆栈是否解决了这个问题？（通常它确实解决了。）
2. 依赖项有多大？（检查打包影响。）
3. 它是否积极维护？（检查最后的提交、未解决的问题。）
4. 它是否有已知漏洞？（`npm audit`）
5. 许可证是什么？（必须与该项目的兼容。）

**规则：** 优先标准库和现有实用程序而不是新依赖项。每个依赖项都是负债。

## 审查清单

```markdown
## 审查：[PR/更改标题]

### 上下文
- [ ] 我理解此更改的作用和原因

### 正确性
- [ ] 更改符合规范/任务要求
- [ ] 处理了边缘情况
- [ ] 处理了错误路径
- [ ] 测试充分覆盖了更改

### 可读性
- [ ] 名称清晰且一致
- [ ] 逻辑简单
- [ ] 无不必要的复杂性

### 架构
- [ ] 遵循现有模式
- [ ] 无不必要的耦合或依赖关系
- [ ] 适当的抽象级别

### 安全性
- [ ] 代码中没有秘密
- [ ] 在边界验证输入
- [ ] 无注入漏洞
- [ ] 到位了身份验证检查
- [ ] 外部数据源被视为不可信

### 性能
- [ ] 无 N+1 模式
- [ ] 无无界操作
- [ ] 列表端点上的分页

### 验证
- [ ] 测试通过
- [ ] 构建成功
- [ ] 已完成手动验证（如果适用）

### 裁决
- [ ] **批准** —— 准备合并
- [ ] **请求更改** —— 必须解决问题
```

## 常见合理化理由

| 合理化理由 | 现实 |
|---|---|
| "它有效，这就足够好了" | 不可读、不安全或架构错误的有效代码会产生复合的债务。 |
| "我写了它，所以我知道它是正确的" | 作者对自己假设视而不见。每个更改都受益于另一双眼睛。 |
| "我们稍后清理它" | 稍后永远不会到来。审查是质量门——使用它。要求在合并之前进行清理，而不是之后。 |
| "AI 生成的代码可能没问题" | AI 代码需要更多审查，而不是更少。即使错误时，它也自信且合理。 |
| "测试通过，所以它很好" | 测试是必要但不充分的。它们不会捕获架构问题、安全问题或可读性担忧。 |

## 危险信号

- PR 在没有任何审查的情况下合并
- 仅检查测试是否通过（忽略其他轴）的审查
- 没有实际审查证据的"LGTM"
- 没有专注安全的审查的安全敏感更改
- "太大而无法正确审查"的大 PR（拆分它们）
- 错误修复 PR 没有回归测试
- 没有严重性标签的审查评论——使不清楚什么是必需 vs 可选
- 接受"我稍后修复它"——它永远不会发生

## 验证

审查完成后：

- [ ] 所有关键问题都已解决
- [ ] 所有重要问题都已解决或有明确推迟的理由
- [ ] 测试通过
- [ ] 构建成功
- [ ] 验证故事有文档记录（更改了什么，如何验证）
