# 弃用和迁移

## 概述

代码是负债，而不是资产。每一行代码都有持续的维护成本——要修复的错误、要更新的依赖项、要应用的安全补丁，以及要让新工程师上手。弃用是删除不再值得保留的代码的纪律，迁移是将用户从旧系统安全移动到新系统的过程。

大多数工程组织擅长构建东西。很少擅长删除它们。这个技能解决了这个差距。

## 使用场景

- 用新系统、API 或库替换旧系统
-  sunsetting 不再需要的功能
- 整合重复的实现
- 删除没有人拥有但每个人都依赖的死代码
- 规划新系统的生命周期（弃用规划在设计时开始）
- 决定是维护遗留系统还是投资迁移

## 核心原则

### 代码是负债

每一行代码都有持续的成本：它需要测试、文档、安全补丁、依赖项更新，以及对附近工作的任何人的心理开销。代码的价值是它提供的功能，而不是代码本身。当相同的功能可以用更少的代码、更少的复杂性或更好的抽象提供时——旧代码应该消失。

### 海勒姆定律使删除变得困难

拥有足够多的用户，每个可观察的行为都会被人依赖——包括错误、时序怪癖和未记录的副作用。这就是为什么弃用需要主动迁移，而不仅仅是公告。当用户依赖替换不复制的行为时，他们不能"只是切换"。

### 弃用规划在设计时开始

在构建新东西时，询问："我们如何在 3 年内删除这个？" 设计时具有干净接口、功能标志和最小表面积的系统比在实现细节处处在处的系统更容易弃用。

## 弃用决策

在弃用任何东西之前，回答这些问题：

```
1. 这个系统是否仍提供独特的价值？
   → 如果是，维护它。如果否，继续。

2. 有多少用户/消费者依赖它？
   → 量化迁移范围。

3. 是否存在替换？
   → 如果没有，首先构建替换。不要在没有替代品的情况下弃用。

4. 每个消费者的迁移成本是多少？
   → 如果可以轻松自动化，就这样做。如果是手动且高投入，请根据维护成本进行权衡。

5. 不弃用的持续维护成本是多少？
   → 安全风险、工程师时间、复杂性的机会成本。
```

## 强制性与咨询性弃用

| 类型 | 何时使用 | 机制 |
|------|-------------|-----------|
| **咨询性** | 迁移是可选的，旧系统稳定 | 警告、文档、提示。用户按照自己的时间表迁移。 |
| **强制性** | 旧系统有安全问题、阻碍进展或维护成本不可持续 | 硬性截止日期。旧系统将在日期 X 删除。提供迁移工具。 |

**默认为咨询性。** 仅在维护成本或风险证明强制迁移合理时才使用强制性。强制性弃用需要提供迁移工具、文档和支持——你不能只是宣布截止日期。

## 迁移过程

### 步骤 1：构建替换

不要在没有工作替代品的情况下弃用。替换必须：

- 覆盖旧系统的所有关键用例
- 有文档和迁移指南
- 在生产中得到验证（不仅仅是"理论上更好"）

### 步骤 2：公告和记录

```markdown
## 弃用通知：OldService

**状态：** 自 2025-03-01 起弃用
**替换：** NewService（见下面的迁移指南）
**删除日期：** 咨询性 —— 还没有硬性截止日期
**原因：** OldService 需要手动扩展且缺乏可观察性。
            NewService 自动处理两者。

### 迁移指南
1. 将 `import { client } from 'old-service'` 替换为 `import { client } from 'new-service'`
2. 更新配置（见下面的示例）
3. 运行迁移验证脚本：`npx migrate-check`
```

### 步骤 3：增量迁移

一次迁移一个消费者，而不是一次全部。对于每个消费者：

```
1. 识别与弃用系统的所有接触点
2. 更新以使用替换
3. 验证行为匹配（测试、集成检查）
4. 删除对旧系统的引用
5. 确认没有回归
```

**搅动规则：** 如果你拥有被弃用的基础设施，你有责任迁移你的用户——或提供不需要迁移的向后兼容更新。不要宣布弃用，然后让用户自己弄清楚。

### 步骤 4：删除旧系统

仅在所有消费者都迁移后：

```
1. 验证零活跃使用（指标、日志、依赖关系分析）
2. 删除代码
3. 删除相关的测试、文档和配置
4. 删除弃用通知
5. 庆祝 —— 删除代码是一项成就
```

## 迁移模式

### 绞杀者模式

并行运行旧系统和新系统。将流量从旧系统增量路由到新系统。当旧系统处理 0% 的流量时，删除它。

```
阶段 1：新系统处理 0%，旧系统处理 100%
阶段 2：新系统处理 10%（金丝雀）
阶段 3：新系统处理 50%
阶段 4：新系统处理 100%，旧系统空闲
阶段 5：删除旧系统
```

### 适配器模式

创建一个适配器，将调用从旧接口转换为新实现。当你迁移后端时，消费者继续使用旧接口。

```typescript
// 适配器：旧接口，新实现
class LegacyTaskService implements OldTaskAPI {
  constructor(private newService: NewTaskService) {}

  // 旧方法签名，委托给新实现
  getTask(id: number): OldTask {
    const task = this.newService.findById(String(id));
    return this.toOldFormat(task);
  }
}
```

### 功能标志迁移

使用功能标志将消费者从旧系统切换到新系统，一次一个：

```typescript
function getTaskService(userId: string): TaskService {
  if (featureFlags.isEnabled('new-task-service', { userId })) {
    return new NewTaskService();
  }
  return new LegacyTaskService();
}
```

## 僵尸代码

僵尸代码是没有人拥有但每个人都依赖的代码。它没有积极维护，没有明确的所有者，并且累积安全漏洞和兼容性问题。迹象：

- 6+ 个月没有提交，但存在活跃的消费者
- 没有分配的所有者或团队
- 没有人修复的失败测试
- 没有人更新的已知漏洞的依赖项
- 引用不再存在的系统的文档

**响应：** 要么分配一个所有者并正确维护它，要么用具体的迁移计划弃用它。僵尸代码不能停留在 limbo 中——它要么得到投资，要么被删除。

## 常见合理化理由

| 合理化理由 | 现实 |
|---|---|
| "它仍然工作，为什么要删除它？" | 没有人在维护的工作代码会累积安全债务和复杂性。维护成本会无声地增长。 |
| "以后可能有人需要它" | 如果以后需要，可以重建。保留未使用的代码"以防万一"比重建成本更高。 |
| "迁移太昂贵" | 将迁移成本与 2-3 年内的持续维护成本进行比较。迁移通常长期更便宜。 |
| "我们会在完成新系统后弃用它" | 弃用规划在设计时开始。到新系统完成时，你会有新的优先事项。现在计划。 |
| "用户会自己迁移" | 他们不会。提供工具、文档和激励——或者自己做迁移（搅动规则）。 |
| "我们可以无限期维护两个系统" | 两个系统做同样的事情是双倍的维护、测试、文档和入门成本。 |

## 危险信号

- 没有可用替换的弃用系统
- 没有迁移工具或文档的弃用公告
- 多年来一直是咨询性但没有进展的"软"弃用
- 没有所有者但有活跃消费者的僵尸代码
- 向弃用系统添加新功能（改为投资替换）
- 在没有衡量当前使用情况下弃用
- 在没有验证零活跃消费者的情况下删除代码

## 验证

完成弃用后：

- [ ] 替换已在生产中得到验证，并涵盖所有关键用例
- [ ] 迁移指南存在，有具体步骤和示例
- [ ] 所有活跃的消费者都已迁移（通过指标/日志验证）
- [ ] 旧代码、测试、文档和配置已完全删除
- [ ] 代码库中没有对弃用系统的引用
- [ ] 弃用通知已删除（它们已达到目的）
