# 增量实现

## 概述

以薄的垂直切片构建 —— 实现一块，测试它，验证它，然后扩展。避免一次实现整个功能。每个增量应该使系统保持在可工作、可测试的状态。这是使大型功能可管理的执行纪律。

## 使用场景

- 实现任何多文件更改
- 从任务分解构建新功能
- 重构现有代码
- 任何你 tempted 在测试之前编写超过 ~100 行代码的时候

**何时不使用：** 范围已经最小的单文件、单函数更改。

## 增量循环

```
┌──────────────────────────────────────┐
│                                      │
│   实现 ──→ 测试 ──→ 验证 ──┐  │
│       ▲                           │  │
│       └──── 提交 ◄─────────────┘  │
│              │                       │
│              ▼                       │
│          下一个切片                  │
│                                      │
└──────────────────────────────────────┘
```

对于每个切片：

1. **实现** 最小完整的功能块
2. **测试** —— 运行测试套件（如果不存在则编写测试）
3. **验证** —— 确认切片按预期工作（测试通过、构建成功、手动检查）
4. **提交** —— 用描述性消息保存你的进度（有关原子提交指导，请参见 `git-workflow-and-versioning`）
5. **移动到下一个切片** —— 向前推进，不要重新开始

## 切片策略

### 垂直切片（首选）

通过堆栈构建一条完整路径：

```
切片 1：创建任务（数据库 + API + 基础 UI）
    → 测试通过，用户可以通过 UI 创建任务

切片 2：列出任务（查询 + API + UI）
    → 测试通过，用户可以看到他们的任务

切片 3：编辑任务（更新 + API + UI）
    → 测试通过，用户可以修改任务

切片 4：删除任务（删除 + API + UI + 确认）
    → 测试通过，完整的 CRUD 完成
```

每个切片都交付可工作的端到端功能。

### 契约优先切片

当后端和前端需要并行开发时：

```
切片 0：定义 API 契约（类型、接口、OpenAPI 规范）
切片 1a：根据契约实现后端 + API 测试
切片 1b：根据匹配契约的模拟数据实现前端
切片 2：集成并端到端测试
```

### 风险优先切片

首先解决风险最高或最不确定的部分：

```
切片 1：证明 WebSocket 连接工作（最高风险）
切片 2：在已验证的连接上构建实时任务更新
切片 3：添加离线支持和重新连接
```

如果切片 1 失败，你在投资切片 2 和 3 之前就会发现它。

## 实现规则

### 规则 0：简单性优先

在编写任何代码之前，询问："什么是最简单有效的方法？"

编写代码后，根据这些检查审查它：
- 这可以用更少的行完成吗？
- 这些抽象是否值得其复杂性？
- 员工工程师会看着这个说"你为什么不直接..."？
- 我是为假设的未来需求构建，还是为当前任务构建？

```
简单性检查：
✗ 用于一个通知的通用 EventBus 和中间件管道
✓ 简单函数调用

✗ 两个类似组件的抽象工厂模式
✓ 两个带有共享实用程序的简单组件

✗ 用于三个表单的配置驱动表单构建器
✓ 三个表单组件
```

三行相似的代码比过早的抽象更好。首先实现朴素的、明显正确的版本。只有在测试证明正确性后才优化。

### 规则 0.5：范围纪律

只触及任务需要的内容。

不要：
- "清理" 你更改旁边的代码
- 重构你未修改的文件中的导入
- 删除你不完全理解的注释
- 添加规范中没有的功能，因为它们"看起来有用"
- 现代化你只读文件中的语法

如果你注意到任务范围之外值得改进的东西，请记下来 —— 不要修复它：

```
注意到但未触及：
- src/utils/format.ts 有一个未使用的导入（与此任务无关）
- 身份验证中间件可以使用更好的错误消息（单独的任务）
→ 要我为这些创建任务吗？
```

### 规则 1：一次一件事

每个增量更改一个逻辑事情。不要混合关注点：

**坏：** 一个添加新组件、重构现有组件和更新构建配置的提交。

**好：** 三个单独的提交 —— 每个更改一个。

### 规则 2：保持可编译

在每个增量之后，项目必须构建并且现有测试必须通过。不要在切片之间将代码库留在损坏状态。

### 规则 3：不完整功能的功能标志

如果一个功能尚未准备好给用户使用，但你需要合并增量：

```typescript
// 进行中工作的功能标志
const ENABLE_TASK_SHARING = process.env.FEATURE_TASK_SHARING === 'true';

if (ENABLE_TASK_SHARING) {
  // 新的共享 UI
}
```

这让你可以将小增量合并到主分支，而不会暴露不完整的工作。

### 规则 4：安全默认值

新代码应该默认为安全、保守的行为：

```typescript
// 安全：默认禁用，选择加入
export function createTask(data: TaskInput, options?: { notify?: boolean }) {
  const shouldNotify = options?.notify ?? false;
  // ...
}
```

### 规则 5：回滚友好

每个增量应该是可独立还原的：

- 附加更改（新文件、新函数）易于还原
- 对现有代码的修改应该是最小且集中的
- 数据库迁移应该有相应的回滚迁移
- 避免在一个提交中删除某些东西并在同一提交中替换它 —— 将它们分开

## 与代理协作

指导代理增量实现时：

```
"让我们实现计划中的任务 3。

从数据库模式更改和 API 端点开始。
还不要触及 UI —— 我们将在下一个增量中做这个。

实现后，运行 `npm test` 和 `npm run build` 来验证
没有东西损坏。"
```

对每个增量明确什么是范围内的，什么是范围外的。

## 增量检查表

在每个增量之后，验证：

- [ ] 更改做一件事并且完全做到
- [ ] 所有现有测试仍然通过（`npm test`）
- [ ] 构建成功（`npm run build`）
- [ ] 类型检查通过（`npx tsc --noEmit`）
- [ ] Linting 通过（`npm run lint`）
- [ ] 新功能按预期工作
- [ ] 更改已用描述性消息提交

## 常见合理化理由

| 合理化理由 | 现实 |
|---|---|
| "我会在最后测试它" | 错误会复合。切片 1 中的错误会使切片 2 -5 出错。测试每个切片。 |
| "一次做完更快" | 它*感觉* 更快，直到某些东西损坏并且你找不到哪一行更改的 500 行导致了它。 |
| "这些更改太小了，不能单独提交" | 小提交是免费的。大提交隐藏错误并使回滚痛苦。 |
| "我会稍后添加功能标志" | 如果功能未完成，它不应该对用户可见。现在就添加标志。 |
| "这个重构足够小，可以包含" | 重构与功能混合使两者都更难审查和调试。分开它们。 |

## 危险信号

- 超过 100 行代码而未运行测试
- 单个增量中的多个不相关更改
- "让我也快速添加这个" 范围扩展
- 跳过测试/验证步骤以更快地移动
- 增量之间的构建或测试损坏
- 累积的大型未提交更改
- 在第三个用例要求之前构建抽象
- 在任务范围之外"趁我在"时触及文件
- 为一次性操作创建新实用程序文件

## 验证

完成一个任务的所有增量后：

- [ ] 每个增量都经过单独测试和提交
- [ ] 完整测试套件通过
- [ ] 构建干净
- [ ] 功能按规范端到端工作
- [ ] 没有未提交的更改保留
