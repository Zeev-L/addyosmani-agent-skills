# 规划与任务分解

## 概述

将工作分解为具有明确验收标准的小型、可验证的任务。好的任务分解是可靠完成工作的代理与产生混乱 mess 的代理之间的区别。每个任务应该足够小，以便在单个集中会话中实现、测试和验证。

## 使用场景

- 你有规范并需要将其分解为可实现的单元
- 任务感觉太大或模糊而无法开始
- 工作需要跨多个代理或会话并行化
- 你需要向人类传达范围
- 实现顺序不明显

**何时不使用：** 范围明显的单文件更改，或规范已包含明确定义的任务时。

## 规划过程

### 步骤 1：进入规划模式

在编写任何代码之前，以只读模式操作：

- 阅读规范和相关的代码库部分
- 识别现有模式和约定
- 映射组件之间的依赖关系
- 注意风险和未知事项

**在规划期间不要编写代码。** 输出是计划文档，而不是实现。

### 步骤 2：识别依赖关系图

映射什么依赖于什么：

```
数据库模式
    │
    ├── API 模型/类型
    │       │
    │       ├── API 端点
    │       │       │
    │       │       └── 前端 API 客户端
    │       │
    │       └── 验证逻辑
    │
    └── 种子数据 / 迁移
```

实现顺序遵循依赖关系图自底向上：首先构建基础。

### 步骤 3：垂直切片

而不是构建所有数据库，然后所有 API，然后所有 UI —— 一次构建一个完整的功能路径：

**坏（水平切片）：**
```
任务 1：构建整个数据库模式
任务 2：构建所有 API 端点
任务 3：构建所有 UI 组件
任务 4：连接所有内容
```

**好（垂直切片）：**
```
任务 1：用户可以创建帐户（用于注册的架构 + API + UI）
任务 2：用户可以登录（身份验证架构 + API + 用于登录的 UI）
任务 3：用户可以创建任务（任务的架构 + API + 用于创建的 UI）
任务 4：用户可以查看任务列表（用于列表视图的查询 + API + UI）
```

每个垂直切片都提供可工作的、可测试的功能。

### 步骤 4：编写任务

每个任务遵循此结构：

```markdown
## 任务 [N]：[简短描述性标题]

**描述：** 一段解释此任务完成内容的段落。

**验收标准：**
- [ ] [具体、可测试的条件]
- [ ] [具体、可测试的条件]

**验证：**
- [ ] 测试通过：`npm test -- --grep "feature-name"`
- [ ] 构建成功：`npm run build`
- [ ] 手动检查：[要验证内容的描述]

**依赖关系：**[此任务依赖的任务编号，或"无"]

**可能触及的文件：**
- `src/path/to/file.ts`
- `tests/path/to/test.ts`

**预估范围：**[小：1-2 个文件 | 中：3-5 个文件 | 大：5+ 个文件]
```

### 步骤 5：排序和检查点

安排任务以便：

1. 满足依赖关系（首先构建基础）
2. 每个任务都使系统保持在可工作状态
3. 每 2-3 个任务后出现验证检查点
4. 高风险任务提前（快速失败）

添加明确的检查点：

```markdown
## 检查点：任务 1-3 之后
- [ ] 所有测试通过
- [ ] 应用程序无错误构建
- [ ] 核心用户流端到端工作
- [ ] 在继续之前与人类一起审查
```

## 任务大小指南

| 大小 | 文件 | 范围 | 示例 |
|------|-------|-------|---------|
| **XS** | 1 | 单函数或配置更改 | 添加验证规则 |
| **S** | 1-2 | 一个组件或端点 | 添加新的 API 端点 |
| **M** | 3-5 | 一个功能切片 | 用户注册流 |
| **L** | 5-8 | 多组件功能 | 带过滤和分页的搜索 |
| **XL** | 8+ | **太大 —— 进一步分解** | — |

如果任务为 L 或更大，则应将其分解为较小的任务。代理在 S 和 M 任务上表现最佳。

**何时进一步分解任务：**
- 将需要超过一个集中会话（大约 2+ 小时代理工作）
- 你无法在 3 个或更少的要点中描述验收标准
- 它触及两个或更多独立子系统（例如，身份验证和计费）
- 你发现自己正在任务标题中写"和"（表示它是两个任务的标志）

## 计划文档模板

```markdown
# 实现计划：[功能/项目名称]

## 概述
[一段关于我们正在构建什么的摘要]

## 架构决策
- [关键决策 1 和理由]
- [关键决策 2 和理由]

## 任务列表

### 阶段 1：基础
- [ ] 任务 1：...
- [ ] 任务 2：...

### 检查点：基础
- [ ] 测试通过，构建干净

### 阶段 2：核心功能
- [ ] 任务 3：...
- [ ] 任务 4：...

### 检查点：核心功能
- [ ] 端到端流工作

### 阶段 3：完善
- [ ] 任务 5：...
- [ ] 任务 6：...

### 检查点：完成
- [ ] 所有验收标准得到满足
- [ ] 准备好审查

## 风险和缓解措施
| 风险 | 影响 | 缓解措施 |
|------|--------|------------|
| [风险] | [高/中/低] | [策略] |

## 开放问题
- [需要人类输入的问题]
```

## 并行化机会

当有多个代理或会话可用时：

- **可安全并行化：** 独立的功能切片、已实现功能的测试、文档
- **必须顺序：** 数据库迁移、共享状态更改、依赖链
- **需要协调：** 共享 API 契约的功能（首先定义契约，然后并行化）

## 常见合理化理由

| 合理化理由 | 现实 |
|---|---|
| "我会边做边弄清楚" | 这就是你最终得到混乱 mess 和返工的方式。10 分钟的规划节省几小时。 |
| "任务是显而易见的" | 无论如何都要写下来。明确的任务反映隐藏的依赖关系和被遗忘的边缘情况。 |
| "规划是开销" | 规划是任务。没有计划的实现只是打字。 |
| "我可以全部记在脑子里" | 上下文窗口是有限的。书面计划可以 surviving 会话边界和压缩。 |

## 危险信号

- 在没有书面任务列表的情况下开始实现
- 说"实现功能"而没有验收标准的任务
- 计划中没有验证步骤
- 所有任务都是 XL 大小
- 任务之间没有检查点
- 未考虑依赖关系顺序

## 验证

在开始实现之前，确认：

- [ ] 每个任务都有验收标准
- [ ] 每个任务都有验证步骤
- [ ] 任务依赖关系已识别并正确排序
- [ ] 没有任务触及超过约 5 个文件
- [ ] 主要阶段之间存在检查点
- [ ] 人类已审查并批准计划
