# 发布与上线#

## 概述#

有信心地发布。目标不仅仅是部署 —— 而是安全地部署，配备监控、回滚计划和清楚理解成功是什么样的。每个发布都应该是可逆的、可观察的和增量的。

## 使用场景#

- 首次将功能部署到生产环境#
- 向用户发布重要更改#
- 迁移数据或基础设施#
- 开放 Beta 或抢先体验计划#
- 任何有风险的部署（所有这些）#

## 功能标志策略#

在功能标志后面发布以将部署与发布解耦：

```typescript
// 功能标志检查
const flags = await getFeatureFlags(userId);

if (flags.taskSharing) {
  // 新功能：任务共享
  return <TaskSharingPanel task={task} />;
}

// 默认：现有行为
return null;
```

**功能标志生命周期：**

```
1. 部署时标志关闭     → 代码在生产中但处于非活动状态#
2. 为团队/beta 启用     → 在生产环境中进行内部测试#
3. 逐步推出          → 5% → 25% → 50% → 100% 的用户#


4. 每个阶段进行监控    → 观察错误率、性能、用户反馈#
5. 清理                 → 完全推出后删除标志和死代码路径#
```

**规则：**
- 每个功能标志都有所有者和过期日期#
- 完全推出后 2 周内清理标志#
- 不要嵌套功能标志（创建指数级组合）#
- 在 CI 中测试两种标志状态（开和关）#

## 分阶段发布#


### 发布序列#

```
1. 部署到预发布环境#
   └── 在预发布环境中进行完整测试套件#
   └── 关键流程的手动冒烟测试#

2. 部署到生产环境（功能标志关闭）#
   └── 验证部署成功（健康检查）#
   └── 检查错误监控（没有新错误）#

3. 为团队启用（内部用户的标志打开）#
   └── 团队在生产环境中使用该功能#
   └── 24 小时监控窗口#


4. 金丝雀发布（为 5% 的用户打开标志）#
   └── 监控错误率、延迟、用户行为#
   └── 比较指标：金丝雀 vs. 基线#
   └── 24-48 小时监控窗口#
   └── 仅在所有阈值通过时推进（见下表）#

5. 逐步增加（25% -> 50% -> 100%）#
   └── 在每个步骤进行相同监控#
   └── 能够在任何时候回滚到之前的百分比#
```


   └── 1 周监控窗口#
   └── 清理功能标志#
```

### 发布决策阈值#

使用这些阈值来决定是否在每个阶段推进、暂停或回滚：

| 指标 | 推进（绿色） | 暂停并调查（黄色） | 回滚（红色） |
|--------|-----------------|-------------------------------|-----------------|
| 错误率 | 在基线的 10% 以内 | 高于基线 10-100% | >2 倍基线 |


| P95 延迟 | 在基线的 20% 以内 | 高于基线 20-50% | >50% 高于基线 |
| 客户端 JS 错误 | 没有新错误类型 | 新错误 <0.1% 的会话 | 新错误 >0.1% 的会话 |
| 业务指标 | 中性或正面 | 下降 <5%（可能是噪音） | 下降 >5% |

### 何时回滚#

如果出现以下情况，立即回滚：
- 错误率增加超过基线的 2 倍#


- P95 延迟增加超过 50%#
- 用户报告的问题激增#
- 检测到数据完整性问题#
- 发现安全漏洞#

## 监控和可观察性#

### 要监控什么#

```
应用程序指标：
├── 错误率（总数和按端点）#
├── 响应时间（p50、p95、p99）#
├── 请求量#
├── 活跃用户数#
└── 关键业务指标（转换、参与度）#

基础设施指标：
├── CPU 和内存利用率#
├── 数据库连接池使用情况#
├── 磁盘空间#
├── 网络延迟#
└── 队列深度（如果适用）#


客户端指标：
├── Core Web Vitals (LCP、INP、CLS)#
├── JavaScript 错误#
├── 客户端视角的 API 错误率#
└── 页面加载时间#
```

## 常见合理化理由#

| 合理化理由 | 现实 |


|---|---|
| "它在预发布环境中工作，在生产环境中也会工作" | 生产环境有不同的数据、流量模式和行为。部署后监控。 |
| "我们不需要为此设置功能标志" | 每个功能都可以从终止开关中受益。即使是"简单" 的更改也可能破坏东西。 |


| "监控是开销" | 没有监控意味着你从用户投诉而不是仪表板中发现问题。 |
| "我们稍后添加监控" | 在发布之前添加它。你无法调试你看不到的东西。 |
| "回滚是承认失败" | 回滚是负责任的工程。发布破坏性的功能是失败。 |
```

## 危险信号#

- 没有回滚计划的部署#
- 生产环境中没有监控或错误报告#
- 大爆炸式发布（一次处理所有事情，没有预发布）#


- 没有所有者或过期日期的功能标志#
- 发布后第一个小时没有人监控部署#
- 生产环境配置凭记忆完成，而不是代码#
- "这是周五下午，让我们发布它"#
