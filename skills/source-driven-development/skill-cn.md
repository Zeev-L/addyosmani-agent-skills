# 源驱动开发#

## 概述#

每个框架特定的代码决策都必须得到官方文档的支持。不要凭记忆实现 —— 验证、引用，并让用户看到你的来源。训练数据会变得过时，API 会被弃用，最佳实践也会演变。此技能确保用户获得他们可以信任的代码，因为每个模式都会追溯到他们可以检查的的有权源。

## 使用场景#

- 用户想要遵循给定框架当前最佳实践代码#
- 构建样板、启动代码或将在项目中复制的模式#
- 用户明确要求有据可查、经过验证或"正确"的实现#
- 实现框架推荐方法很重要的功能（表单、路由、数据获取、状态管理、身份验证）#
- 审查或改进使用框架特定模式的代码#
- 每当你准备凭记忆编写框架特定代码时#

**何时不使用：**


- 正确性不依赖于特定版本（重命名变量、修复错字、移动文件）#
- 在所有版本中工作方式相同的纯逻辑（循环、条件、数据结构）#
- 用户明确想要速度而不是验证（"只管快速完成它"）#

## 过程#

```
检测 ──→ 获取 ──→ 实现 ──→ 引用#
  │          │           │            │#
  ▼          ▼           ▼            ▼#
 什么       获取      遵循      显示你的#
 技术栈？    相关      文档化    来源#
            文档      模式#
```

### 步骤 1：检测技术栈和版本#

读取项目的依赖文件以识别确切的版本：

```
package.json    → Node/React/Vue/Angular/Svelte#
composer.json   → PHP/Symfony/Laravel#
requirements.txt / pyproject.toml → Python/Django/Flask#
go.mod          → Go#
Cargo.toml      → Rust#
Gemfile         → Ruby/Rails#
```

明确地说明你发现了什么：

```
检测到技术栈：
- React 19.1.0 (来自 package.json)
- Vite 6.2.0#
- Tailwind CSS 4.0.3#
→ 正在为相关模式获取官方文档。
```

如果版本缺失或模棱两可，**询问用户**。不要猜测 —— 版本决定了哪些模式是正确的。

### 步骤 2：获取官方文档#

为你正在实现的功能获取特定文档页面。不是主页，不是完整文档 —— 是相关页面。

**源层次结构（按权威顺序）：**

| 优先级 | 来源 | 示例 |
|----------|--------|---------|
| 1 | 官方文档 | react.dev、docs.djangoproject.com、symfony.com/doc |
| 2 | 官方博客 / 变更日志 | react.dev/blog、nextjs.org/blog |
| 3 | Web 标准参考 | MDN、web.dev、html.spec.whatwg.org |
| 4 | 浏览器/运行时兼容性 | caniuse.com、node.green |

**不是权威性的 —— 永远不要作为主源引用：**


- Stack Overflow 答案#
- 博客文章或教程（即使是受欢迎的那些）#
- AI 生成的文档或摘要#
- 你自己的训练数据（这恰恰是关键点 —— 验证它）#

**在获取时要精确：**

```
坏：获取 React 主页#
好：获取 react.dev/reference/react/useActionState#

坏：搜索"django 身份验证最佳实践"#
好：获取 docs.djangoproject.com/en/6.0/topics/auth/#
```

获取后，提取关键模式并注意任何弃用警告或迁移指南。

当官方源相互冲突时（例如，迁移指南与 API 参考冲突），向用户反映差异并验证哪个模式实际适用于检测到的版本。

### 步骤 3：遵循文档化模式实现#

编写与文档显示内容匹配的代码：

- 使用文档中的 API 签名，而不是凭记忆#
- 如果文档显示了执行某事的新方法，请使用新方法#
- 如果文档弃用某个模式，请不要使用弃用版本#
- 如果文档未涵盖某些内容，请将其标记为未验证#

**当文档与现有项目代码冲突时：**

```
检测到冲突：
现有代码库对表单加载状态使用 useState，
但 React 19 文档推荐使用 useActionState 处理此模式。
（来源：react.dev/reference/react/useActionState)#

选项：
A) 使用现代模式（useActionState）—— 与当前文档一致#
B) 匹配现有代码（useState）—— 与代码库一致#
你更喜欢哪种方法？
```

反映冲突。不要默默地选择一个。

### 步骤 4：引用你的来源#

每个框架特定模式都获得引用。用户必须能够验证每个决策。

**在代码注释中：**

```typescript
// 使用 useActionState 的 React 19 表单处理
// 来源：https://react.dev/reference/react/useActionState#usage
const [state, formAction, isPending] = useActionState(submitOrder, initialState);
```

**在对话中：**

```
我正在对表单提交状态使用 useActionState 而不是手动 useState。
React 19 用这个钩子替换了手动 isPending/setIsPending 模式。

来源：https://react.dev/blog/2024/12/05/react-19#actions
"useTransition 现在支持异步函数 [...] 自动处理挂起状态"
```

**引用规则：**

- 完整的 URL，而不是缩短的#
- 尽可能使用带有锚点的深层链接（例如 `/useActionState#usage` 而不是 `/useActionState`）—— 锚点比顶级页面更能经受文档重构#
- 当非显而易见的决策得到支持时，请引用相关段落#
- 推荐平台功能时包括浏览器/运行时支持数据#
- 如果无法找到模式的文档，请明确说明：#

```
未验证：我无法找到此模式的官方文档。
这基于训练数据，可能已经过时。
在生产中使用之前进行验证。
```

对无法验证的事情保持诚实比虚假的信心更有价值。

## 常见合理化理由#

| 合理化理由 | 现实 |
|---|---|
| "我对这个 API 很有信心" | 信心不是证据。训练数据包含看起来正确但在当前版本中破坏的过时模式。验证。 |
| "获取文档浪费令牌" | 产生幻觉的 API 会浪费更多。用户调试一小时，然后发现函数签名已更改。一次获取可防止数小时的返工。 |
| "文档不会有我要的东西" | 如果文档没有涵盖它，那就是有价值的信息 —— 该模式可能不是官方推荐的。 |
| "我只提一下它可能过时" | 免责声明没有帮助。要么验证并引用，要么将其明确标记为未验证。逃避是最糟糕的选择。 |
| "这是一个简单的任务，不需要检查" | 具有错误模式的简单任务会成为模板。用户在发现现代方法存在之前，会将你弃用的表单处理程序复制到十个组件中。 |

## 危险信号#

- 在没有检查该版本的文档的情况下编写框架特定代码#
- 使用"我相信"或"我认为"关于 API 而不是引用源#
- 在不知道适用于哪个版本的情况下实现模式#
- 引用 Stack Overflow 或博客文章而不是官方文档#
- 使用在训练数据中出现的弃用 API#
- 在实现之前不要读取 `package.json` / 依赖文件#
- 交付没有对框架特定决策的源引用的代码#
- 在只有一个页面相关时获取整个文档站点#

## 验证#

使用源驱动开发实现后：

- [ ] 从依赖文件中识别了框架和库版本#
- [ ] 为框架特定模式获取了官方文档#
- [ ] 所有源都是官方文档，而不是博客文章或训练数据#
- [ ] 代码遵循当前版本文档中显示的模式#
- [ ] 非平凡决策包括带有完整 URL 的源引用#
- [ ] 没有使用弃用的 API（根据迁移指南进行检查）#
- [ ] 文档与现有代码之间的冲突已向用户反映#
- [ ] 任何无法验证的事情都被明确标记为未验证#
