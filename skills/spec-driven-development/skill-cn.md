# 规格驱动开发 (Spec-Driven Development)

## 概述

在编写任何代码之前，先编写结构化的规格说明。规格说明是你与人类工程师之间共享的真相来源——它定义了我们在构建什么、为什么构建，以及如何知道它已完成。没有规格说明的代码就是猜测。

## 何时使用

- 启动新项目或新功能
- 需求模糊或不完整
- 变更涉及多个文件或模块
- 即将做出架构决策
- 任务实施需要超过30分钟

**何时不使用：** 单行修复、拼写更正，或需求明确且独立完整的变更。

## 门控工作流

规格驱动开发有四个阶段。在当前阶段验证通过之前，不要进入下一阶段。

```
规格说明 ──→ 计划 ──→ 任务 ──→ 实施
   │          │        │          │
   ▼          ▼        ▼          ▼
 人工审核   人工审核  人工审核    人工审核
```

### 阶段1：规格说明 (Specify)

从高层愿景开始。向人类提出澄清性问题，直到需求具体化。

**立即揭示假设。** 在编写任何规格说明内容之前，列出你的假设：

```
我正在做的假设：
1. 这是一个 Web 应用（不是原生移动应用）
2. 认证使用基于会话的 Cookie（不是 JWT）
3. 数据库是 PostgreSQL（基于现有的 Prisma schema）
4. 我们仅针对现代浏览器（不支持 IE11）
→ 现在纠正我，否则我将按这些假设进行。
```

不要默默地填补模糊的需求。规格说明的全部目的就是在代码编写*之前*揭示误解——假设是最危险的理解偏差形式。

**编写涵盖以下六个核心领域的规格说明文档：**

1. **目标** — 我们在构建什么？为什么？用户是谁？成功是什么样的？

2. **命令** — 带参数的完整可执行命令，不仅仅是工具名称。
   ```
   构建：npm run build
   测试：npm test -- --coverage
    lint：npm run lint --fix
   开发：npm run dev
   ```

3. **项目结构** — 源代码在哪里，测试在哪里，文档在哪里。
   ```
   src/           → 应用源代码
   src/components → React 组件
   src/lib        → 共享工具函数
   tests/         → 单元测试和集成测试
   e2e/           → 端到端测试
   docs/          → 文档
   ```

4. **代码风格** — 一个展示你风格的真实代码片段胜过三段描述它的文字。包括命名约定、格式化规则和优质输出示例。

5. **测试策略** — 使用什么框架，测试放在哪里，覆盖率预期，哪些测试级别针对哪些关注点。

6. **边界** — 三层系统：
   - **始终执行：** 提交前运行测试，遵循命名约定，验证输入
   - **先询问：** 数据库 schema 变更，添加依赖，更改 CI 配置
   - **绝不执行：** 提交密钥，编辑 vendor 目录，未经批准删除失败的测试

**规格说明模板：**

```markdown
# 规格说明：[项目/功能名称]

## 目标
[我们在构建什么以及为什么。用户故事或验收标准。]

## 技术栈
[框架、语言、带版本的关键依赖]

## 命令
[构建、测试、lint、开发 — 完整命令]

## 项目结构
[带说明的目录布局]

## 代码风格
[示例代码片段 + 关键约定]

## 测试策略
[框架、测试位置、覆盖率要求、测试级别]

## 边界
- 始终：[...]
- 先询问：[...]
- 绝不：[...]

## 成功标准
[我们如何知道这已完成 — 具体的、可测试的条件]

## 未决问题
[任何需要人工输入的未解决问题]
```

**将指令重构为成功标准。** 当收到模糊需求时，将它们转化为具体条件：

```
需求："让仪表盘更快"

重构后的成功标准：
- 在 4G 连接下仪表盘 LCP < 2.5秒
- 初始数据加载在 < 500毫秒内完成
- 加载期间无布局偏移（CLS < 0.1）
→ 这些是正确的目标吗？
```

这让你能够循环、重试，并朝着明确的目标解决问题，而不是猜测"更快"意味着什么。

### 阶段2：计划 (Plan)

有了经过验证的规格说明，生成技术实施计划：

1. 识别主要组件及其依赖关系
2. 确定实施顺序（必须先构建什么）
3. 注意风险和缓解策略
4. 识别可以并行构建的内容 vs 必须顺序执行的内容
5. 定义阶段之间的验证检查点

计划应该是可审核的：人类应该能够阅读它并说"是，这是正确的方法"或"不，更改 X"。

### 阶段3：任务 (Tasks)

将计划分解为离散的、可实施的任务：

- 每个任务应该可以在单次专注会话中完成
- 每个任务都有明确的验收标准
- 每个任务都包含一个验证步骤（测试、构建、手动检查）
- 任务按依赖关系排序，而不是按感知的重要性
- 任何任务都不应要求更改超过约5个文件

**任务模板：**
```markdown
- [ ] 任务：[描述]
  - 验收：[完成时必须为真的内容]
  - 验证：[如何确认 — 测试命令、构建、手动检查]
  - 文件：[将触及哪些文件]
```

### 阶段4：实施 (Implement)

按照 `incremental-implementation` 和 `test-driven-development` 技能一次执行一个任务。使用 `context-engineering` 在每一步加载正确的规格说明部分和源文件，而不是用整个规格说明淹没代理。

## 保持规格说明活跃

规格说明是一个活文档，不是一次性产物：

- **当决策变更时更新** — 如果你发现数据模型需要更改，先更新规格说明，然后实施。
- **当范围变更时更新** — 添加或削减的功能应反映在规格说明中。
- **提交规格说明** — 规格说明应该和代码一起纳入版本控制。
- **在 PR 中引用规格说明** — 链接回每个 PR 实施的规格说明部分。

## 常见合理化借口

| 合理化借口 | 现实 |
|---|---|
| "这很简单，我不需要规格说明" | 简单的任务不需要*长*规格说明，但它们仍然需要验收标准。两行的规格说明也可以。 |
| "我会在编码后编写规格说明" | 那是文档，不是规格说明。规格说明的价值在于在代码*之前*强制明确性。 |
| "规格说明会拖慢我们的速度" | 15分钟的规格说明可以防止数小时的返工。15分钟的水fall胜过15小时的调试。 |
| "需求反正会变更" | 这就是为什么规格说明是一个活文档。过时的规格说明仍然比没有规格说明好。 |
| "用户知道他们想要什么" | 即使明确的请求也有隐含的假设。规格说明揭示了这些假设。 |

## 危险信号

- 在没有任何书面需求的情况下开始编写代码
- 在澄清"完成"的含义之前询问"我应该开始构建吗？"
- 实施任何规格说明或任务列表中未提及的功能
- 做出架构决策而不记录它们
- 因为"构建什么很明显"而跳过规格说明

## 验证

在实施之前，确认：

- [ ] 规格说明涵盖所有六个核心领域
- [ ] 人类已经审核并批准了规格说明
- [ ] 成功标准具体且可测试
- [ ] 边界（始终/先询问/绝不）已定义
- [ ] 规格说明已保存到存储库中的文件
